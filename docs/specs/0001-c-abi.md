## Invariants

- The v1 surface (`LMV_ABI_VERSION = 1`) MUST be exactly these eight functions and no
  others: `lmv_abi_version`, `lmv_create`, `lmv_free`, `lmv_push_samples`,
  `lmv_attach_window`, `lmv_render`, `lmv_resize`, `lmv_cycle_scene`. (ADR-0003,
  `core/src/ffi.rs`)
- `LMV_ABI_VERSION` MUST change **only** when the shape of the `extern "C"` surface changes
  (a signature, a function, an error code) â€” never as a side effect of an application version
  bump. It is a separate axis from the app version (ADR-0005). (ADR-0003)
- The C header `core/include/lmv_core.h` MUST match `core/src/ffi.rs` signature-for-signature;
  the shim compiles against the header separately from the Rust crate, so a drift is a
  link/runtime error, not a compile error. (ADR-0003)
- Every pointer crossing the boundary MUST be validated where it enters the core (null checks,
  handle validity); the hot path downstream trusts it. Validate at the boundary, trust inside.
  (CLAUDE.md non-negotiables)
- `lmv_push_samples` MUST NOT allocate, block, lock a contended mutex, log, or do file I/O â€”
  it runs on the caller's audio-delivery thread. It hands samples to the ring and returns.
  (CLAUDE.md; see [0002-ring-determinism](0002-ring-determinism.md))
- Widening the surface (a ninth function, a changed signature) is an ADR-worthy event, not a
  casual edit. The seam stays minimal: create / free / push / attach / render / resize /
  cycle-scene. (ADR-0001, ADR-0003)

## Scenarios

- WHEN the C++ shim calls `lmv_abi_version()` THEN it returns `1`, letting the shim refuse a
  core whose ABI it was not built against.
- WHEN `lmv_create` succeeds THEN it returns an opaque non-null handle that every other call
  (except `lmv_abi_version`) takes as its first argument; WHEN it fails THEN it returns null
  and no handle is leaked.
- WHEN `lmv_push_samples` is called with a valid handle and a sample buffer THEN the samples
  are copied into the lock-free ring and the call returns immediately, without touching the
  render path.
- WHEN `lmv_push_samples` is called with a null handle or null buffer THEN it is rejected at
  the boundary (no deref of the invalid pointer), not passed inward.
- WHEN `lmv_free` is called on a handle THEN all core resources for that handle are released
  and the handle MUST NOT be used again.
- WHEN the `extern "C"` surface must change shape THEN `LMV_ABI_VERSION` is bumped and a new
  ADR supersedes ADR-0003 â€” the app version (ADR-0005) is untouched by this.

## Known gaps / honest nulls

- **No automated test coverage yet.** The C++ shim is not built in CI, and no in-crate Rust
  FFI test exercises `lmv_create` / `lmv_push_samples` / `lmv_free`. This is a known open gap
  (plans README); Miri over the extracted ring (Plan 0005) will cover the ring/pointer
  handling once it runs, but a create/push/free FFI test remains unassigned.
- This spec covers the *contract shape and boundary discipline*, not the per-call payload
  semantics of rendering (frame timing, surface format) â€” those live with the render/scene
  subsystem and the wgpu layer.
